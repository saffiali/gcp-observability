GCP Implementation Playbook: Deployment Guide

A step-by-step technical guide for implementing this exact Anomaly Forecasting system in your GCP environment (gke-demos-363017).

1

Upgrade Cloud Logging to Log Analytics

To run BigQuery ML `ARIMA_PLUS` models directly on ServiceNow logs, you must first upgrade your logging bucket (e.g. _Default) to use Log Analytics. This links the bucket directly to BigQuery for SQL-based analysis.

# Upgrade the _Default logging bucket to support Log Analytics
gcloud logging buckets update _Default \
    --location=global \
    --enable-analytics

# Verify the BigQuery linked dataset name
gcloud logging buckets describe _Default \
    --location=global
2

Define Log-Based Metrics (for ServiceNow Logs)

If ServiceNow logs are sent via GKE stdout or Cloud Logging agent, create Log-Based Metrics to extract incident counts and partition by country and site location.

# Create log-based metric to count incidents
gcloud logging metrics create servicenow_incident_count \
    --description="Counts the daily ServiceNow incident volumes" \
    --log-filter="jsonPayload.event=\"incident\" AND jsonPayload.message:\"ServiceNow incident created\"" \
    --value-extractor="INT64(jsonPayload.incident_count)"
3

Train the BigQuery ML ARIMA_PLUS Model

Run the training SQL directly in the BigQuery interface of gke-demos-363017. The ARIMA_PLUS model will automatically handle seasonality, holiday trends, missing data imputation, and outlier scrubbing.

CREATE OR REPLACE MODEL `gke-demos-363017.servicenow_forecasts.incident_model`
OPTIONS(
  model_type = 'ARIMA_PLUS',
  time_series_timestamp_col = 'timestamp',
  time_series_data_col = 'incident_count',
  time_series_id_col = ['country', 'site'],
  holiday_region = 'GLOBAL',
  clean_spikes_and_dips = TRUE,
  adjust_step_changes = TRUE
) AS
SELECT 
  TIMESTAMP(FORMAT_TIMESTAMP('%Y-%m-%d 00:00:00', timestamp)) as timestamp,
  jsonPayload.country,
  jsonPayload.site,
  COUNT(1) as incident_count
FROM `gke-demos-363017.global._Default._AllLogs`
WHERE logName LIKE '%servicenow%'
  AND timestamp >= TIMESTAMP_SUB(CURRENT_TIMESTAMP(), INTERVAL 90 DAY)
GROUP BY 1, 2, 3;
4

Schedule Predictive Alerting Policies

Using Google Cloud Monitoring, deploy a Predictive Alert policy on your infrastructure metrics. This triggers alerts when a capacity limit is forecast to be exceeded within 24 hours.

# Apply Terraform configuration for predictive alerting
terraform init
terraform plan -var="project_id=gke-demos-363017"
terraform apply -auto-approve

Why GCP Native beats ELK for ML

  • No Infrastructure Overhead: Kibana ML requires dedicated Elasticsearch ML nodes ($), which must be sized, scaled, and managed. BigQuery ML is 100% serverless, zero-ops, and executes in seconds.
  • Global Scale via Monarch: Google Managed Prometheus metrics are stored globally in Monarch, enabling extremely fast queries (PromQL/MQL) without index scaling issues or high cluster costs.
  • Multi-Series Ingestion Natively: Using BQML's time_series_id_col, you can forecast millions of timeseries (such as incidents grouped by 100 sites and 50 countries) in a single, simple SQL call.
  • Advanced Time-Series Modeling out of the box: GCP's ARIMA_PLUS model integrates automatic holiday effects (for 50+ countries), handles daylight savings, imputes missing timestamps, and handles level shifts automatically.
